# RafaDex v1.1 Polish — Design Spec

**Date:** 2026-07-21
**Status:** Approved design, pending implementation plan

## What

A post-launch polish pass driven by real usage: Rafael started using RafaDex on the
parent's iPhone (v1 shipped 2026-07-21, live at `https://atavaresm.github.io/rafadex/`)
and four gaps surfaced immediately. This spec covers all four as one bundled release,
since they're small, independent, and touch the same two screens (detail, type grid)
plus the audio layer.

1. Swipe-to-navigate on the detail screen (in addition to the existing ‹ › arrows).
2. A way to stop the long flavor-text narration mid-read without leaving the Pokémon.
3. A fix for pt-BR text-to-speech mispronouncing many Pokémon names.
4. Richer type-grid cards: Pokédex number, generation, type badge(s), and a "power"
   (total base stats) figure, shown as a discreet corner overlay — the big colorful
   type icon remains the primary thing Rafael actually navigates by.

## Non-goals

- Visual redesign / "more modern" look — explicitly deferred by the parent to a later
  round.
- Any change to the arrows' existing navigation semantics (wrap-around, context
  scoping to type/favorites) — swipe is additive, not a replacement.
- Full Gen 2–9 pronunciation coverage at launch — only Gen 1 (151) is seeded now;
  later generations grow the same table incrementally as mispronunciations are found,
  mirroring how `i18n-dex.js`'s hand-translated categories/flavors grew.
- Any change to the ⚡ cry button — it's a short audio clip, not speech; no stop
  control needed or requested.

## 1. Swipe-to-navigate (detail screen)

`renderDetail()`'s `.detail` box gains `touchstart`/`touchend` listeners measuring
horizontal displacement. A swipe left triggers the same handler as the › (next)
button; a swipe right triggers the same handler as the ‹ (prev) button — same
`contextIds` wrap-around, same context scoping (type world, favorites, or full dex).
Threshold: 50px horizontal displacement, with a vertical-displacement guard (if the
vertical delta exceeds the horizontal delta, treat it as a scroll gesture and ignore
it, so a diagonal drag never fires an accidental navigation). The ‹ › buttons stay
exactly as they are — swipe is purely additive.

## 2. Narration stop-toggle (🔊 and 📖)

`Sound` gains a way to know it is currently speaking a specific utterance, and the
detail screen's sound buttons become toggles: tapping 🔊 or 📖 while idle starts
speech (as today); tapping the **same button again while it is speaking** stops it
immediately, like a play/pause control. Both buttons visually indicate "currently
speaking" (a highlighted/pressed style) so a child sees which one to tap again to
stop. This applies to both the short name (🔊) and the long narration (📖) — same
mechanism, since both go through `Sound.speak()`.

Implementation shape: `Sound.speak(text, { onStart, onEnd })` (or an equivalent
speaking-state getter) so `app.js` can track "am I the button currently speaking"
per detail-screen render and toggle between calling `Sound.speak()` and
`Sound.stopSpeech()` on click, without needing a global speaking flag that survives
across re-renders (each `renderDetail()` call creates fresh button state, same
pattern already used for the heart/favorite toggle).

## 3. Pronunciation fix

Root cause: iOS's pt-BR voice mispronounces many Pokémon names because they're
English-coined words with phonetic patterns pt-BR text-to-speech doesn't map
correctly (e.g. "-ard" endings, "th", certain vowel clusters).

Fix: a new **pipeline-generated** field, `speak`, alongside the existing `cat` and
`flavor` fields in `data/dex.js` — an optional pt-BR phonetic respelling used *only*
when calling `Sound.speak()` for that Pokémon's name. The **displayed** name
(`mon.name`) never changes; only the string handed to the speech synthesizer does.
Entries without an override fall back to `mon.name`, exactly like `i18n-dex.js`'s
existing fallback-to-English pattern for category/flavor.

Source of truth: a new hand-curated file, `pronounce-dex.js`, living directly in the
**RafaDex** repo (not upstream in `pokedex`) — mirroring `i18n-dex.js`'s shape,
containing `{id: "phonetic respelling"}` only for names that need it. It belongs in
RafaDex specifically because pronunciation is a RafaDex-only concern: the upstream
`pokedex` web dashboard has no text-to-speech feature at all, so there's nothing to
share by pushing this data upstream — keeping it local also means `build.py` doesn't
need to write back to the read-only `pokedex` dependency.

Seeding: before this ships, listen to all 151 Gen 1 names read by a real pt-BR voice
(browser `speechSynthesis`, same voice-selection logic as the app) and write a
respelling for every one that comes out wrong. This covers the bulk of Rafael's
day-to-day use immediately (Gen 1 is the precached, most-played set). Later
generations get entries added the same way, incrementally, as mispronunciations are
reported — not all 1025 need to be pre-tested now.

Call sites affected: the 🔊 button (`Sound.speak(mon.name)`) and the 📖 button's name
portion (`Sound.speak(`${mon.name}. ${mon.cat}. ${mon.flavor}`)`) both switch to using
`mon.speak || mon.name` for the spoken name, while `mon.cat`/`mon.flavor` are
unaffected (they're already natural pt-BR text, not proper nouns).

## 4. Richer type-grid cards

Each `.mon-card` in `renderType()`'s grid keeps its current big thumbnail (primary
navigation surface) and adds a small, visually secondary overlay in one corner
showing, together: the Pokédex number (`#025` style), the generation, this
Pokémon's type badge(s) (relevant for dual-type Pokémon, since the grid itself is
scoped to a single type already — e.g. Charizard shown in the Fire world still shows
a small Flying badge too), and the "power" figure — `stats.total` from the source
data (already computed upstream, e.g. Charizard = 534) — styled like a Pokémon
trading card's power number. This is corner/secondary treatment, not a redesign of
the card's primary visual language; the type-tinted background and big sprite stay
exactly as they are.

Data already available: `gen` and `types` are already in `data/dex.js`; `stats.total`
is present in the upstream `pokedex` project's `pokemon.json` but **not currently
carried into `data/dex.js`** — the pipeline's `buildDexEntries()` needs to add a
`power` field (`mon["stats"]["total"]`) to each entry.

## Data flow (delta from v1)

```
pokedex/data/pokemon.json (stats.total) ─┐
pokedex/i18n-dex.js (existing, read-only) ├─build.py──▶ data/dex.js (+power field)
rafadex/pronounce-dex.js (new, hand-curated,┘            (+speak field, optional)
  lives in this repo, not upstream)
```

## Error handling

- Swipe: a tap that isn't a clear horizontal drag (vertical scroll, a tap on a
  button inside `.detail`) must never fire navigation — the vertical-displacement
  guard covers scrolling; swipe listeners are on the `.detail` container, not on
  interactive children, so button taps are unaffected.
- Narration toggle: if `Sound.speak()` is called for a new utterance while another
  is already playing (e.g. tapping 🔊 then immediately 📖), the new call must cancel
  the previous one first (already true of the existing `Sound.speak()` internals)
  and correctly reset both buttons' visual "speaking" state — no stuck highlight.
- Pronunciation: missing `speak` field is not an error, just the existing
  name-fallback behavior.
- Card power/number/gen/type badges: these are static data already validated by the
  v1 pipeline's completeness checks; no new failure mode introduced.

## Testing

- Pipeline: extend `tests/test_build.py` to cover the new `power` field
  (`buildDexEntries` includes `stats.total` as `power`) and the `speak` field's
  fallback behavior (present → used verbatim; absent → falls back to `name`),
  mirroring the existing i18n-merge test pattern.
- Frontend: live verification per house rule — swipe gesture, stop-toggle on both
  sound buttons (tap-to-start, tap-again-to-stop, visual state), and the enriched
  card's four new data points, checked in a real browser; then a real-device pass
  on the iPhone (the same Task 13 gate) for the pronunciation fix specifically,
  since that's the one issue that can only be judged by ear on the actual target
  voice.

## Implementation phases (for the plan)

1. Pipeline: add `power` field; add `pronounce-dex.js` merge producing the `speak`
   field (start with an empty table — seeding happens in phase 4).
2. Detail screen: swipe gesture + narration stop-toggle (🔊 and 📖).
3. Type-grid card: number/generation/type-badge(s)/power corner overlay.
4. Pronunciation seeding: listen to all 151 Gen 1 names via pt-BR `speechSynthesis`,
   write `pronounce-dex.js` entries for the ones that sound wrong, rebuild, verify
   the specific corrected names live.
5. Real-iPhone re-verification pass (swipe, stop-toggle, card content, and — the one
   that matters most for this round — pronunciation on the actual iOS voice).
