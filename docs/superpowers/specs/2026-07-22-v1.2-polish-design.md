# RafaDex v1.2 Polish — Design Spec

**Date:** 2026-07-22
**Status:** Approved design, pending implementation plan

## What

A second post-launch polish round, covering the backlog items surfaced during v1.1's
real-device verification and deploy:

1. Fix a real layout bug: the evolution strip overflows the viewport on
   many-branch Pokémon (Eevee, 8 branches), causing unwanted page-level horizontal
   scroll on the real iPhone.
2. Restore the type-grid's scroll position when returning from a detail card.
3. Make the detail screen's header (back button + title) sticky and taller, so it's
   always reachable during scroll and never obscures content.
4. A parent-facing name search (not for Rafael — he doesn't read yet; this is a
   quick-lookup tool for the parent).
5. Fix the root cause of the v1.1 deploy finding: the service worker's cache
   version was a hardcoded literal a human has to remember to bump, so an
   already-installed app didn't pick up v1.1's fixes automatically. This makes it
   automatic going forward.

## Non-goals

- No visual redesign — still deferred by the parent to a later round.
- No change to Rafael-facing navigation (type worlds, favorites, game) beyond the
  bug fix in item 1 — search is parent-only, reached via the existing discreet
  gear-icon pattern, not a new Home-screen element aimed at Rafael.
- No redesign of the evolution strip's *interaction model* (still a horizontal
  scroll strip) — item 1 fixes the actual bug (page overflow), it does not redesign
  the UX for many-branch chains, which the parent did not ask for.

## 1. Evolution strip overflow fix

**Root cause:** `.evo-strip` is a flex item inside `.detail` (itself a flex
container). Flex items default to `min-width: auto`, meaning a flex item that is
itself a flex container refuses to shrink below its content's intrinsic width unless
told otherwise. Eevee's chain (8 thumbnails + 7 arrow glyphs) has an intrinsic width
of roughly 810px, far exceeding the ~500px available inside `.detail`/`#app`. Because
nothing in the ancestor chain (`html`, `body`, `#app`, `.detail`) clips horizontal
overflow, the oversized flex item pushes the whole page wider, producing the
"estoura a tela, gera scroll lateral indesejado" the parent observed on the real
iPhone. This didn't reproduce during v1's desktop-Chrome verification of Eevee,
which is why it shipped unnoticed — flexbox overflow-clipping behavior around this
exact `min-width: auto` case is a known cross-browser inconsistency.

**Fix:** add `min-width: 0` to `.evo-strip`, so it is now allowed to shrink to its
parent's available width — the strip's own pre-existing `overflow-x: auto` then
correctly takes over, scrolling its contents internally instead of blowing out the
page. As a defensive safety net (matching a lesson already learned in the sibling
`pokedex` project), add `overflow-x: hidden` to `html, body` — this guarantees no
future layout mistake can ever again cause page-level horizontal scroll, regardless
of its root cause.

## 2. Scroll position restore

A single in-memory map, keyed by hash string (e.g. `#type/fire`), stores the
scroll-Y position captured the moment the route changes away from it. `renderRoute()`
saves the *previous* hash's scroll position before dispatching to the new screen,
renders the new screen, then scrolls to that hash's previously-saved position (or 0,
if this is the first visit) — restoring exactly where the parent/Rafael left off
when returning to a type world (or any screen) via the back button, swipe-back
context, or any other navigation. This is intentionally screen-agnostic (not special
cased to type grids only) since the mechanism is equally simple either way and it's
one less special case to reason about.

## 3. Sticky, taller detail header

`.topbar` becomes `position: sticky; top: 0` with a solid `var(--bg)` background and
increased vertical padding (taller). It stays pinned to the top of the viewport
during scroll, so the back button and title (and, transitively, the type emoji next
to the title) are always visible and tappable, and scrolling content passes cleanly
underneath rather than through it. This applies to `.topbar` everywhere it's used
(type world, detail, game) since it's a single shared component — the request came
from the detail screen specifically but there is no reason to special-case it there.

## 4. Parent-facing name search

A second discreet icon (🔍), placed next to the existing ⚙️ gear on the Home screen,
toggles a small panel (same interaction pattern as the download panel): a text input
plus a live-filtered result list. Filtering is case-insensitive and
diacritic-insensitive substring matching against each Pokémon's `name` field,
re-computed on every keystroke (cheap — 1025 entries, plain string filtering), capped
to the first 20 matches (ordered by id) to keep the panel short for a broad query.
Each result row shows a small thumbnail + name; tapping one sets `contextIds` to the
full dex (consistent with how a directly-loaded `#dex/N` link already behaves — no
type or favorites scoping) and navigates straight to that Pokémon's detail screen.

## 5. Automatic service worker cache versioning

Instead of a hardcoded `const VERSION = "v1"` in `sw.js` that a human must remember
to bump on every deploy that changes cached content (the exact mistake that shipped
v1.1's fixes into a cache namespace indistinguishable from v1's, per the real-device
finding), `build.py` now writes a small generated file, `sw-version.js`, containing a
build timestamp constant. `sw.js` imports it via the same `importScripts` mechanism
it already uses for `precache.js`, and uses that constant as its cache-version
string. Since browsers detect service worker updates by byte-comparing the SW's main
script *and* everything it imports via `importScripts`, a changed timestamp in
`sw-version.js` is enough to make the browser recognize a real update on every build
that actually changes shipped content — no manual bump, no repeat of this failure
mode.

## Data flow (delta from v1.1)

```
build.py (at build time) ──▶ sw-version.js (new, timestamp constant)
sw.js ──importScripts──▶ precache.js + sw-version.js ──▶ cache name uses the timestamp
```

## Error handling

- Scroll restore: a hash never visited before has no map entry — falls back to 0,
  identical to today's behavior. No error path; a plain object lookup with `|| 0`.
- Evolution strip: the fix is purely corrective CSS; no new failure mode.
- Search: an empty or no-match query simply renders an empty result list, not an
  error — no crash, no placeholder text needed beyond "no results" being an empty
  panel (kept minimal since this is a parent tool, not a kid-facing empty state that
  needs hand-holding).
- SW versioning: if `sw-version.js` somehow fails to exist (e.g. a manual partial
  deploy), `sw.js`'s `importScripts` would throw at SW startup — acceptable given
  this file is generated by the same pipeline run that produces every other
  committed asset; it is never absent in practice, matching how `precache.js` is
  already treated as always-present.

## Testing

- Pipeline: extend `tests/test_build.py` to cover the new `sw-version.js`-writing
  step (asserts the file is written with a `SW_BUILD` constant of the expected
  shape).
- Frontend: live verification per house rule — evolution strip on Eevee (no page
  horizontal scroll, internal scroll still works), scroll restore (scroll a grid,
  open a detail, go back, verify position), sticky header (scroll a long grid,
  confirm header stays pinned and legible), search (type a partial name, confirm
  filtered results, tap one, confirm correct navigation), and the service worker fix
  needs a **before/after real update check**: deploy this version, confirm a
  browser/device that already had an old service worker installed actually receives
  the update this time (this is the one regression test that directly targets the
  bug this task fixes).

## Implementation phases (for the plan)

1. Evolution strip overflow fix (CSS only, smallest and highest-priority — it's a
   real bug affecting current use).
2. Scroll restore + sticky header (both touch `renderRoute`/`.topbar`, small).
3. Parent-facing search panel.
4. Automatic SW cache versioning (pipeline change).
5. Deploy and verify — including the before/after SW-update check.
